# 版本管理 GIT -- Fork

## GIT

### 步骤1：Fork 安装与配置

#### 1. Fork 的安装

官网下载地址：https://git-fork.com/

- 找到下载的文件（版本使用最新的即可）
- 双击打开安装包，一路默认安装即可

#### 2. Fork 界面介绍

![图片1.png](pic%2F%E5%9B%BE%E7%89%871.png)

##### 重要概念

| 概念 | 说明 |
| --- | --- |
| **工作区（Unstaged）** | 电脑本地文件夹里的文件，**所有指向工作区的操作都有可能产生冲突** |
| **暂存区（Staged）** | 从工作区里选择的变更内容，哪些是要提交的就放到暂存区 |
| **本地仓库（Branches）** | 将暂存区里的文件进行提交后，进入本地的版本库里（仅自己可见） |
| **远端仓库（Remotes）** | 所有人共享的一个公共仓库 |

##### 界面介绍
**操作区**（4个常用按钮）：

| 按钮 | 功能 |
| --- | --- |
| **Fetch** | 把远端仓库的内容同步到本地仓库里，但不会修改电脑里的任何文件（工作区不受影响），此时你可以看到本地分支上跟远端分支的差异 |
| **Pull** | 把远端仓库的内容，更新到本地仓库，并修改电脑里的相关文件（工作区被修改）。即更新。此时，工作区里如果有相关的文件在本地也做了修改，则会产生冲突。因此，在 Pull 之前，需要将工作区（Unstaged）里的修改 discard 掉 |
| **Push** | 把本地仓库的内容，更新到远端仓库。推送的是通过暂存区提交的所有文件，而工作区里其他内容不会同步到远端。因为在 Push 之前，可能其他人也在远端相同分支提交过，因此，**一定要先 Pull 更新别人的内容，再 Push 自己的** |
| **Stash** | 本地修改的内容，不想丢弃掉，但因为想同步远端不能保留在工作区，此时可以放到一个叫储藏区（Stashes）的地方，等下次想用的时候，再去储藏区取出来 |

**分支状态**：
- 显示了当前是哪个分支，以及跟远端分支的差异
- 上箭头 ↑ 表示本地有多少个 Commit 未推送到远端仓库
- 下箭头 ↓ 表示有多少个 Commit 未更新到本地仓库

**本地查看**（2种类型）：
- **Local Changes**：查看工作区的变更，以及暂存区有哪些文件
- **All Commits**：查看当前分支的所有提交记录

**本地分支**：
- Branches 可以看做是本地仓库，下面的分支列表记录了本地仓库有哪些分支
- 用来进行分支管理，点击分支后，本地查看方式会立即切换到 All Commits 状态

**远程分支**：
- Remotes 可以看做是远端仓库，下面记录了远端有哪些分支

**本地文件变更**：
- Unstaged 可以看做是工作区，查看本地文件有哪些发生了变更但还没有进入仓库

**暂存区**：
- Staged 可以看做是暂存区，选择需要进入仓库的文件

**提交操作**：
- 按照规范输入本次提交的内容，并点击提交后可以将暂存区的文件加入到本地仓库

![图片2.png](pic%2F%E5%9B%BE%E7%89%872.png)

#### 3. 如何克隆工程

1. 选择文件 → Clone
2. 填入工程地址、本地文件夹路径
3. 等待远端文件下载完成即可

![图片3.png](pic%2F%E5%9B%BE%E7%89%873.png)

#### 4. 如何拉取分支、切换分支、创建分支

**拉取/切换分支**：
- 在 Remotes 远程仓库或 Branches 本地仓库里找到或搜索到你需要的分支
- 远程仓库里如果没有，点击一次 Fetch 操作
- **右键选择 Checkout 或者直接双击该分支**

> **注意**：切换分支会直接修改工作区文件，因此需要将本地修改丢弃掉。

- 如果在切换时勾选了 `Stash and reapply local changes`，会自动将本地修改进行贮藏，并在切换后重新放回工作区，但有可能产生冲突
- 出现冲突警告时，需要先把列出的文件进行处理（丢弃或自己贮藏）

**创建分支**：
- 在提交历史上，选择一个节点 → 右键选择 `New Branch...`，可以基于该提交创建新分支
- 在本地仓库里选择一个分支 → 右键选择 `New Branch...`，则基于该分支（最新提交）创建新分支

#### 5. 如何提交一个 Commit

1. 将要提交的文件从工作区拖到（或双击）暂存区
2. 输入符合规范的提交内容，点击 Commit 即可

![图片4.png](pic%2F%E5%9B%BE%E7%89%874.png)

**工作区显示模式**：

| 模式 | 说明 |
| --- | --- |
| 树形结构（默认） | 按文件夹折叠展示，防止内容过多不好找 |
| View as List | 每个文件一行展示 |
| View as Combined List | 路径和文件名分开展示 |
| Hide Untracked Files | 隐藏不在版控里的（新增）文件 |
| Show Ignored Files | 展示忽略文件（一般用不到） |

#### 6. 如何使用 Pull 更新别人的提交

1. 点击 Pull 按钮
2. 选择远端的目标分支（一般不用改）
3. **勾选 `Rebase instead of merge` 选项**
4. 点击 Pull 即可

> **重要**：Pull 会直接更新工作区，因此**工作区、暂存区都不可以有未提交的内容**。处理方式：
> - 提交到一个临时 commit，Pull 之后再通过 reset 把临时 commit 删除
> - 直接 discard 丢弃掉
> - 使用 stash 保存（见第9部分）

#### 7. 如何使用 Push 把自己的提交更新到远程

- 完成 commit 后，点击 Push 按钮进行推送

![图片5.png](pic%2F%E5%9B%BE%E7%89%875.png)

> **注意：Push 选项中的 2 个复选框不要勾选！**
> - **Force push** 是强推，会导致其他人的提交丢失。仅当分支开发结束要合入主干时，确定没有人再提交，且本地已更新到最新提交时，rebase 后进行一次强推。

**Push 常见报错**：
- 最常见的是远程还有更新而你本地没有先拉一次 → 查看报错信息底部是否有 `git pull` 的 hint 提示
- 另一种是没有推送权限（主干、release 等保护分支）→ 需要基于该分支拉一个开发分支，完成后通过 Merge Request 进行合入

#### 9. 如何使用 Stash 保存本地修改

- 点击 Stash 按钮，会将所有工作区、暂存区内容直接放到贮藏区
- 提示输入贮藏记录的名字（也可以不输入，自动生成）
- 如果不想全部保存，可以多选需要的文件进行保存
- 贮藏区在 Fork 左边栏最下方的 **Stashes** 区域
- 右键选择 **Apply** 即可将该记录恢复到工作区和暂存区

> **注意**：Apply 时有可能产生冲突。

#### 10. 常见报错问题解决方案

Git 操作中遇到报错莫要慌，**报错信息里一般都有报错原因和解决办法的建议**。如果英文看不懂，可以找个翻译工具查一查。报错情况就几种，遇到多了就记住了。

#### 11. 遇到了冲突如何解决？

- Git 是版本管理软件，可以提示冲突，但本身不是用来解决冲突的
- 对于代码文件：内置的 diff 工具可以帮助对比差异，方便解决冲突
- 对于二进制文件：diff 工具并不友好，**最好的方式是沟通，大家协商好谁先谁后**

![图片6.png](pic%2F%E5%9B%BE%E7%89%876.png)

**冲突解决界面**：左边表示本地的，右边表示远程（别人）的

**非代码冲突的处理方式**（Pull 使用 rebase 时，相当于先用远程内容再用自己的）：
- 可以直接选择的（如图片）：哪个分支是新的就用哪个
- 不能直接选择的：先用他人的，然后把自己的修改在他人版本上再做一次
- 如果忘记自己的修改内容或想做备份：使用 **Abort** 按钮放弃本次更新，去本地手动保存后再重新拉取

![图片7.png](pic%2F%E5%9B%BE%E7%89%877.png)

#### 12. 分支开发 rebase 主干后无法推送

由于 rebase 会打破原来的提交记录，远端虽然有之前的提交内容，但 rebase 后相同的提交跟远端已不再是同一个 commit 了。因此，这里需要使用一次**强推（Force Push）**。

#### 13. 使用 Rebase 代替 Merge

##### 效果对比

![merge_vs_rebase.svg](pic%2Fmerge_vs_rebase.svg)

| 对比项 | 合并（Merge） | 变基（Rebase） |
| --- | --- | --- |
| 排序方式 | 按 commit 时间排序 | 按合入顺序排序 |
| 合入节点 | 保留合入节点 | 合入时重新创建 commit 和合入点 |
| 可读性 | 提交内容混合，较难看出哪些是原来就有的 | 提交内容清晰，可通过 tag 看到上一次发布后有哪些变更 |
